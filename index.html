Introduction to DevOps

9 HOURS · FOUNDATIONS

🤔 What is DevOps? ▼
  • DevOps is a combination of "Development" (Dev) and "Operations" (Ops) — two teams that traditionally worked separately.
  • It's a culture, philosophy, and set of practices that brings these two teams together to deliver software faster and more reliably.
  • Think of it like this: Dev team writes code, Ops team deploys & maintains it. DevOps removes the wall between them.
  • It emphasizes collaboration, communication, and automation throughout the software delivery pipeline.
  • DevOps is NOT a tool — it's a mindset and a methodology that uses tools to achieve its goals.
  • The ultimate goal: ship code from developer's laptop to production servers quickly, safely, and repeatedly.
  • Companies like Netflix, Amazon, and Google use DevOps to deploy code thousands of times per day.
📜 History of DevOps ▼
  • Before DevOps (pre-2008): Dev and Ops teams worked in silos — developers wrote code, threw it "over the wall" to operations, and conflicts arose constantly.
  • 2008: Patrick Debois and Andrew Shafer coined the concept during the Agile Conference, frustrated with the Dev vs Ops conflict.
  • 2009: The first "DevOpsDays" conference was held in Ghent, Belgium — the term "DevOps" was born on Twitter (hashtag #devops).
  • 2010s: Tools like Jenkins, Docker, Puppet, Chef, and Ansible emerged to support DevOps practices.
  • The "Phoenix Project" book (2013) popularized DevOps thinking through a business novel format.
  • Today, DevOps is a standard industry practice adopted by companies of all sizes worldwide.
🎯 DevOps Main Objectives ▼
  • Faster Delivery: Deploy software updates in days or hours instead of months — "release early, release often."
  • Improved Collaboration: Break silos between Dev, QA, and Ops teams through shared responsibilities and tools.
  • Continuous Improvement: Always monitor, measure, and improve the software development and delivery process.
  • Higher Quality: Automated testing and integration catch bugs early before they reach production.
  • Reduced Risk: Smaller, frequent releases mean smaller changes — easier to find and fix issues.
  • Automation: Automate repetitive tasks (testing, building, deploying) to free humans for creative work.
  • Customer Satisfaction: Deliver value to customers faster and respond quickly to their feedback.
🌊 Waterfall Model ▼
  • Waterfall is a traditional, linear software development model where each phase completes before the next begins.
  • Phases in order: Requirements → Design → Implementation → Testing → Deployment → Maintenance.
  • Like a waterfall — water flows down, you can't go back up. Once a phase is done, you rarely revisit it.
  • Problem: If you find a bug in testing, going back to fix design-level issues is very expensive and slow.
  • Releases happen very infrequently — months or even years between software versions.
  • DevOps replaced Waterfall thinking because it's too slow and rigid for modern software needs.
  • Good for projects with very fixed, unchanging requirements (e.g., defense systems, medical devices).
🔄 Agile Model ▼
  • Agile is an iterative software development methodology — work is done in small cycles called "Sprints" (usually 2 weeks).
  • Each sprint produces a working piece of software that can be shown to customers for feedback.
  • Agile is flexible — requirements can change between sprints based on customer feedback or market conditions.
  • Core Agile values: Individuals over processes, working software over docs, customer collaboration over contracts, responding to change over following a plan.
  • DevOps extends Agile: Agile focuses on development speed; DevOps adds automation of deployment and operations.
  • Popular Agile frameworks: Scrum (sprints, stand-ups, backlogs) and Kanban (visual board, continuous flow).
  • Agile + DevOps together = code written fast AND deployed fast with quality.
🔁 Continuous Integration & Deployment (CI/CD) ▼
  • Continuous Integration (CI): Developers merge their code changes into a shared repository multiple times a day. Each merge triggers automated tests.
  • CI prevents "integration hell" — when developers work separately for too long and merging becomes a nightmare.
  • Continuous Delivery (CD): After CI passes, code is automatically prepared and ready to be deployed to production at any time.
  • Continuous Deployment: Takes it further — code automatically deploys to production after passing all tests, no human approval needed.
  • Jenkins is the most popular CI/CD tool — it automates the build, test, and deploy pipeline.
  • CI/CD pipeline stages: Code → Build → Test → Stage → Deploy. Each stage is automated.
  • Benefits: Faster feedback on bugs, less risky releases, and always having a deployable version of code.
📦 Containers & Virtual Development (Docker, Vagrant) ▼
  • The classic developer problem: "It works on my machine!" — Containers solve this by packaging app + dependencies together.
  • Docker creates lightweight containers — isolated environments that run consistently on any machine.
  • Containers share the host OS kernel (unlike VMs which have their own full OS), making them faster and lighter.
  • Vagrant is a tool for managing Virtual Machines (VMs) — it automates the setup of complete VM environments using a "Vagrantfile" config.
  • Difference: Docker = containers (lightweight, seconds to start). Vagrant = full VMs (heavier, minutes to start).
  • Both help create reproducible development environments — every team member works in identical conditions.
  • Docker is the dominant technology today; Vagrant is older and mainly used for full VM-based workflows.
⚙️ Configuration Management (Ansible, Puppet, Chef) ▼
  • Configuration Management (CM) automates the setup and maintenance of servers — no more manually SSHing into 100 servers!
  • Ansible: Simple, agentless (no software needed on remote servers), uses YAML files called "Playbooks." Easiest to learn. Push-based.
  • Puppet: Uses its own language (Puppet DSL), requires an agent installed on each server, pull-based (servers pull config from master).
  • Chef: Uses Ruby, very powerful and flexible, requires agents. Uses "Recipes" and "Cookbooks" as config files. Steeper learning curve.
  • All three ensure servers are always in the desired state — if someone manually changes a config, CM tools fix it automatically.
  • Idempotency: Running CM tools multiple times gives the same result — no unintended side effects.
  • CM tools are essential for infrastructure at scale — managing hundreds or thousands of servers consistently.
02

Version Control — GIT

9 HOURS · SOURCE CONTROL

✨ GIT Features ▼
  • Git is a distributed version control system — every developer has a full copy of the entire repository history on their machine.
  • Tracks every change to every file — who changed what, when, and why (via commit messages).
  • Branching is extremely fast and lightweight in Git — create a branch in milliseconds.
  • Works offline — you can commit, branch, and view history without internet. Only push/pull needs a connection.
  • Distributed: No single point of failure. If a central server crashes, any developer's copy can restore it.
  • Data integrity: Git uses SHA-1 hashes to identify every commit — impossible to silently corrupt history.
  • Free and open-source, created by Linus Torvalds (creator of Linux) in 2005 for managing the Linux kernel source code.
🌲 3-Tree Architecture ▼
  • Git has 3 "trees" (areas) that files move through. Understanding this is the KEY to understanding Git.
  • Tree 1 — Working Directory: Where you actually edit files. This is your normal filesystem.
  • Tree 2 — Staging Area (Index): A preparation zone. You choose which changes to include in the next commit using `git add`.
  • Tree 3 — Repository (.git folder): The permanent history. Once committed with `git commit`, changes are saved forever.
  • Workflow: Edit files (Working Dir) → Stage changes (git add → Staging Area) → Commit (git commit → Repository).
  • The Staging Area lets you craft precise commits — e.g., you changed 5 files but only want to commit 3 of them.
  • Think of it like: Draft (Working) → Review (Staging) → Publish (Repository).
📤 GIT Clone / Commit / Push ▼
  • git clone [URL]: Downloads a complete copy of a remote repository to your local machine, including all history and branches.
  • git add [file]: Moves changes from Working Directory to the Staging Area. Use `git add .` to stage all changes.
  • git commit -m "message": Saves staged changes permanently into the local repository with a descriptive message.
  • git push origin main: Uploads your local commits to the remote repository (e.g., GitHub) so others can see them.
  • git status: Shows which files are modified, staged, or untracked — your most-used Git command.
  • git log: Shows the history of all commits — who committed what and when.
  • The typical daily workflow: pull latest → make changes → add → commit → push. Repeat!
🔀 GIT Rebase & Merge ▼
  • Both Merge and Rebase integrate changes from one branch into another, but they do it differently.
  • git merge: Combines two branches by creating a new "merge commit." Preserves the exact history of both branches.
  • git rebase: Moves your branch's commits onto the tip of another branch — creates a linear, cleaner history.
  • Analogy: Merge = "We worked separately, then merged our work." Rebase = "I pretend I started my work after yours."
  • Merge is safer for public/shared branches. Rebase is great for cleaning up local commits before merging.
  • Golden rule: Never rebase commits that have already been pushed to a shared repository — it rewrites history and causes chaos.
  • Merge conflicts occur when both branches changed the same line. Git marks the conflict and you resolve it manually.
📦 GIT Stash ▼
  • Stash is like a temporary drawer — it saves your unfinished work so you can switch to another task urgently.
  • git stash: Takes all your uncommitted changes and puts them aside, leaving your working directory clean.
  • git stash pop: Brings your stashed changes back and removes them from the stash.
  • git stash apply: Brings changes back but keeps them in the stash too (useful for applying to multiple branches).
  • git stash list: Shows all your saved stashes — you can have multiple stashes.
  • Scenario: You're halfway through a feature when your boss says "fix this bug NOW!" Stash your work, fix the bug, pop your stash, continue.
  • Stash does NOT save untracked (new) files by default. Use `git stash -u` to include untracked files.
⏪ GIT Reset, Checkout ▼
  • git checkout [branch]: Switches to a different branch, updating your working directory to match that branch.
  • git checkout [file]: Discards all changes to a file in your working directory — restores it to last commit state.
  • git reset --soft HEAD~1: Undoes the last commit but keeps changes staged. Commit is gone, code is saved.
  • git reset --mixed HEAD~1 (default): Undoes the last commit and unstages changes. Code stays in working directory.
  • git reset --hard HEAD~1: DANGER — Completely destroys the last commit AND all changes. Cannot be undone easily.
  • HEAD is a pointer to your current position in Git history. HEAD~1 means "one commit before current."
  • Modern Git uses `git switch` and `git restore` instead of the overloaded `git checkout` command.
⬇️ GIT Clone, Fetch, Pull ▼
  • git clone: One-time operation to copy an entire remote repository to your machine for the first time.
  • git fetch: Downloads new data from the remote repository but does NOT merge it into your local branches. Safe operation.
  • git pull: Downloads AND immediately merges remote changes into your current branch. It's git fetch + git merge combined.
  • When to use fetch: You want to see what changed remotely before deciding to merge. Good for reviewing changes first.
  • When to use pull: You're confident and want to update your branch with the latest remote changes immediately.
  • git pull --rebase: Fetches and rebases instead of merging — keeps a cleaner, linear history.
  • Always pull before you push to avoid conflicts — get the latest code before adding yours.
🐙 GitHub Projects & Management ▼
  • GitHub is a cloud hosting service for Git repositories — it adds a web UI, collaboration features, and project management on top of Git.
  • Pull Requests (PRs): A GitHub feature where you propose merging your branch. Team members review, comment, and approve before merging.
  • Issues: GitHub's built-in bug tracker and task manager — teams create issues for bugs, features, and tasks.
  • GitHub Projects: A Kanban-style board built into GitHub to track issues and PRs through stages (To Do → In Progress → Done).
  • Forking: Copying someone else's public repository to your own GitHub account to experiment or contribute.
  • GitHub Actions: Built-in CI/CD — you can define workflows (YAML files) that run automatically on events like push or PR.
  • Organizations and Teams allow managing multiple repositories and controlling who can access and modify what.

Continuous Integration — Jenkins

9 HOURS · CI/CD AUTOMATION

🤖 Introduction to Jenkins ▼
  • Jenkins is a free, open-source automation server that helps automate building, testing, and deploying software.
  • Written in Java, Jenkins runs as a web server — you access it through a browser on port 8080 by default.
  • Jenkins is the most popular CI/CD tool in the world — used by millions of developers and companies.
  • It acts like a robot that watches your Git repository. When you push code, Jenkins automatically builds and tests it.
  • Jenkins has a massive plugin ecosystem (1800+ plugins) that lets it integrate with almost any tool.
  • Originally forked from "Hudson" in 2011 when Oracle acquired Sun Microsystems and the community moved to Jenkins.
  • Jenkins can be installed on any OS (Linux, Windows, Mac) or run as a Docker container.
🔧 Configure & Manage Jenkins ▼
  • Jenkins is configured through its web UI at http://localhost:8080 — first-time setup uses an auto-generated admin password.
  • Global Configuration: Set up JDK, Maven, Git, and other tool paths that Jenkins will use for builds.
  • Jenkins home directory stores all jobs, build history, logs, and plugin data — back it up regularly!
  • Manage Jenkins → Configure System: Set global settings like email notifications, environment variables, and node configurations.
  • Jenkins uses a "Master" (now called Controller) to manage everything and distributes work to "Agent" (Slave) nodes.
  • System Log in Jenkins: View Jenkins internal logs to debug issues with the server itself.
  • Jenkins can be managed via CLI (jenkins-cli.jar) or REST API for scripting and automation.
⏰ Scheduling Build Jobs (Poll SCM & Build Periodically) ▼
  • Jenkins can trigger builds automatically in multiple ways — you don't have to click "Build" manually every time.
  • Poll SCM: Jenkins periodically checks your Git repository for new commits. If new code is found, it triggers a build. Uses cron syntax.
  • Build Periodically: Triggers a build at scheduled times regardless of whether code changed — like a cron job. e.g., `0 2 * * *` = build at 2 AM daily.
  • Cron syntax: `* * * * *` = (minute, hour, day of month, month, day of week). H symbol = Jenkins picks a smart time to spread load.
  • Webhook Trigger (preferred): Git services like GitHub send a notification to Jenkins when code is pushed — instant triggering, no polling delay.
  • Poll SCM is less efficient because Jenkins wastes resources checking for changes even if nothing happened.
  • Example Poll SCM: `H/5 * * * *` = check for changes every 5 minutes.
📋 Different Types of Jenkins Jobs ▼
  • Freestyle Project: The simplest Jenkins job — configure build steps through the UI, point to a repo, and add shell/batch commands. Best for beginners.
  • Maven Project: Special job type that understands Maven's build lifecycle — automatically parses pom.xml and runs Maven goals.
  • Pipeline: Modern approach — define your entire build, test, and deploy pipeline as code in a "Jenkinsfile." Most powerful and recommended.
  • Multibranch Pipeline: Automatically creates pipeline jobs for each branch in your repository. Great for feature branch workflows.
  • GitHub Organization: Scans an entire GitHub organization and creates multibranch pipelines for all repos automatically.
  • Folder: Not a job type but an organizational tool — groups related jobs together in the Jenkins UI.
  • Matrix Project (Multi-Configuration): Runs the same build with different combinations — e.g., test on Python 3.8 AND 3.9 AND 3.10 simultaneously.
🔗 Jenkins Build Pipeline (Parent/Child & Sequential) ▼
  • A Build Pipeline chains multiple Jenkins jobs together — output of one job becomes input of the next.
  • Parent-Child Builds: A parent (upstream) job triggers child (downstream) jobs. E.g., Build job triggers Test job which triggers Deploy job.
  • Sequential Builds: Jobs run one after another in sequence — Stage 1 must pass before Stage 2 starts.
  • Parallel Builds: Multiple jobs run simultaneously — e.g., run unit tests AND integration tests at the same time to save time.
  • Jenkins Pipeline plugin provides the `Build Pipeline View` — a visual dashboard showing job chains and their statuses.
  • Jenkinsfile defines the pipeline as code: `pipeline { stages { stage('Build') { ... } stage('Test') { ... } stage('Deploy') { ... } } }`
  • If any stage fails, the pipeline stops and marks the build as FAILED — preventing bad code from reaching production.
🖥️ Jenkins Master & Slave Node Configuration ▼
  • Jenkins Master (Controller): The main Jenkins server — manages jobs, schedules builds, distributes work, and stores results. Should NOT run builds itself in large setups.
  • Jenkins Slave (Agent): Remote machines that actually execute the build jobs assigned by the master.
  • Why use agents? Distribute workload, run builds on different OS (e.g., build iOS app on Mac agent), run in parallel, scale horizontally.
  • Connection methods: SSH (Linux agents), JNLP/Java Web Start (Windows agents), or Jenkins Docker agents.
  • Labels: Tag agents with labels (e.g., "linux", "docker", "java11") and assign jobs to specific label types.
  • In Jenkinsfile: `agent { label 'linux' }` — run this pipeline only on agents labeled "linux."
  • Cloud agents (AWS, GCP, Kubernetes): Spin up agents on-demand, run the build, then terminate — cost-efficient for variable workloads.
🔒 Securing Jenkins ▼
  • By default, Jenkins may allow anyone to do anything — you MUST secure it before exposing it to the internet.
  • Authentication: Who can log in? Options include Jenkins own user database, LDAP (company directory), GitHub OAuth, or SSO.
  • Authorization: What can logged-in users do? Matrix-based security lets you grant specific permissions (read, build, configure, admin) per user/group.
  • Role-Based Access Control (RBAC): Use the "Role-based Authorization Strategy" plugin to define roles (developer, QA, admin) and assign them to users.
  • Confidentiality: Use HTTPS (SSL/TLS) to encrypt traffic to Jenkins — never run Jenkins over plain HTTP in production.
  • Credentials Store: Jenkins has a secure vault for storing passwords, API keys, SSH keys, and certificates — never hardcode secrets in Jenkinsfiles!
  • Creating Users: Manage Jenkins → Manage Users → Create User. Always disable the anonymous read access option.
🧩 Jenkins Plugins ▼
  • Plugins extend Jenkins' capabilities — Jenkins itself is minimal; plugins add almost all functionality.
  • Installing Plugins: Manage Jenkins → Manage Plugins → Available tab → Search → Install without restart (or with restart for some).
  • SCM Plugin: Integrates Jenkins with version control systems like Git, SVN. The "Git Plugin" is essential for almost every Jenkins setup.
  • Build Plugins: Maven Integration Plugin, Gradle Plugin, Ant Plugin — let Jenkins understand and execute various build tools.
  • Test Plugins: JUnit Plugin displays test results in a beautiful UI. Coverage plugins show code coverage reports.
  • Pipeline Plugin (Blue Ocean): Provides the modern Pipeline Job type and a beautiful visual pipeline editor.
  • Popular essential plugins: Git, Pipeline, Blue Ocean, Credentials, Email Extension, Slack Notification, Docker, Kubernetes.

Build Tool — Maven

9 HOURS · BUILD AUTOMATION

🏗️ Maven Introduction & Installation ▼
  • Maven is a build automation tool primarily used for Java projects — it handles compilation, testing, packaging, and dependency management.
  • Think of Maven as a "project manager" — it knows how to build your project from source code to a final deployable artifact (JAR/WAR).
  • Installation: Download Apache Maven, extract it, set MAVEN_HOME environment variable, add bin/ to PATH. Verify with `mvn --version`.
  • Maven uses a standard project structure — `src/main/java` for code, `src/test/java` for tests, `target/` for build output.
  • Convention over Configuration: Maven follows strict conventions so you write less configuration. If you follow its structure, it "just works."
  • Maven requires Java (JDK) to be installed — it's a Java tool that builds Java projects.
  • Alternative to Maven: Gradle (newer, more flexible), Ant (older, manual). Maven is most widely used in enterprise Java.
📄 Maven POM (pom.xml) ▼
  • POM = Project Object Model — the pom.xml file is the heart of every Maven project. It defines everything about the project.
  • Every pom.xml must have: groupId (organization), artifactId (project name), and version (e.g., 1.0.0). This trio uniquely identifies your project in the Maven world.
  • groupId: Your organization's identifier, usually reverse domain name — e.g., `com.google`, `org.apache`, `com.mycompany`.
  • artifactId: The name of your specific project/module — e.g., `my-web-app`, `user-service`.
  • version: Identifies the version — e.g., `1.0.0`, `2.3.1`, or `1.0-SNAPSHOT` for work in progress.
  • Dependencies are declared in pom.xml — Maven automatically downloads them from the internet (Maven Central repository).
  • Super POM: Every pom.xml inherits from Maven's built-in "Super POM" which provides default configurations — you only override what you need.
🔄 Maven Build Lifecycle ▼
  • Maven has a fixed build lifecycle — a sequence of phases that execute in order. You can't skip phases (earlier phases always run first).
  • Key phases in order: validate → compile → test → package → verify → install → deploy.
  • validate: Checks pom.xml is correct and all necessary info is available.
  • compile: Compiles Java source code in src/main/java into bytecode in target/classes.
  • test: Runs unit tests in src/test/java using a test framework like JUnit. Build fails if tests fail.
  • package: Packages compiled code into a distributable format — JAR file for libraries, WAR for web apps.
  • install: Copies the packaged artifact to your local .m2 repository so other local Maven projects can use it as a dependency.
  • deploy: Uploads the artifact to a remote repository (like Nexus or Artifactory) for team sharing. Used in CI/CD.
📦 Maven Repositories (.m2, Global, Central) ▼
  • Maven has a 3-tier repository system for finding and caching dependencies.
  • Local Repository (.m2): A folder on your machine (at ~/.m2/repository). First place Maven checks for dependencies. Acts as a cache.
  • Remote/Central Repository: Maven Central (https://repo.maven.apache.org) — the default public repository with millions of open-source Java libraries.
  • Corporate/Private Repository: Tools like Nexus Repository or JFrog Artifactory — companies host their own artifacts here for internal sharing.
  • Lookup order: Local (.m2) → Private/Corporate Repo → Maven Central. If found locally, no download needed (faster builds).
  • settings.xml in ~/.m2/ lets you configure proxy, mirrors, and credentials for accessing private repositories.
  • `mvn dependency:resolve` downloads all declared dependencies to your local .m2 cache.
🔗 Maven Dependencies & Snapshot ▼
  • Dependencies in pom.xml declare external libraries your project needs — Maven downloads them automatically. No more manual JAR downloads!
  • Each dependency is identified by groupId + artifactId + version — same coordinates used for your own project.
  • Dependency Scope: `compile` (default, always available), `test` (only for testing, e.g. JUnit), `provided` (available at runtime but not packaged, e.g. Servlet API), `runtime` (needed to run but not compile).
  • Transitive Dependencies: If Library A depends on Library B, Maven automatically downloads B too — you only declare direct dependencies.
  • SNAPSHOT: A version ending in `-SNAPSHOT` (e.g., `1.0-SNAPSHOT`) means it's in active development and can change. Maven re-downloads it periodically.
  • RELEASE: A stable, finalized version. Once released, it NEVER changes. `1.0.0` is a release version.
  • Dependency conflicts: When two libraries need different versions of the same dependency, Maven uses "nearest wins" or "first declaration" rules.
🔌 Maven Plugins ▼
  • Maven itself is just a framework — all actual build actions (compile, test, package) are performed by Plugins.
  • Plugins are bound to lifecycle phases — e.g., the Compiler Plugin runs during the `compile` phase automatically.
  • maven-compiler-plugin: Compiles Java code. You configure Java version here (e.g., source/target = 11 or 17).
  • maven-surefire-plugin: Runs unit tests during the `test` phase. Works with JUnit and TestNG.
  • maven-jar-plugin: Creates a JAR file during the `package` phase.
  • maven-war-plugin: Creates a WAR file for web applications during the `package` phase.
  • Plugins have Goals — specific tasks the plugin can perform. e.g., `mvn compiler:compile` runs only the compile goal of the compiler plugin.

Docker

9 HOURS · CONTAINERIZATION

🐋 What is Docker & Docker Images? ▼
  • Docker is a platform for building, shipping, and running applications in isolated environments called containers.
  • Docker Image: A read-only template with everything needed to run an app — code, runtime, libraries, environment variables, config files.
  • Images are built in layers — each instruction in a Dockerfile adds a layer. Layers are cached and reused for faster builds.
  • Think of an Image like a class in OOP — it's a blueprint. A Container is like an object — a running instance of the blueprint.
  • Images are immutable — you never modify a running container to change the image. Instead you build a new image.
  • Getting Images: docker pull [image-name] downloads from Docker Hub. e.g., `docker pull nginx` downloads the Nginx web server image.
  • docker images: Lists all images on your machine. docker rmi [image]: Removes an image. docker inspect [image]: Shows detailed image metadata.
⚙️ Docker Installation & Engine ▼
  • Docker Desktop: Easy installer for Windows and Mac that includes Docker Engine, CLI, and a GUI dashboard.
  • Linux: Install Docker Engine directly via package manager. `sudo apt install docker.io` on Ubuntu, then `sudo systemctl start docker`.
  • Docker Engine is the core runtime — it's a client-server application with a daemon (dockerd), a REST API, and the CLI client (docker).
  • Docker Daemon (dockerd): The background service that manages all Docker objects (images, containers, networks, volumes). Runs as root.
  • Docker CLI: The command-line tool you use to interact with Docker. Commands like `docker run`, `docker build`, `docker ps`.
  • Verify installation: `docker --version` shows version. `docker run hello-world` runs a test container — confirms everything works.
  • Docker uses Linux kernel features (namespaces, cgroups) for isolation — on Windows/Mac, Docker Desktop runs a lightweight Linux VM.
📦 Working with Docker Containers ▼
  • Container: A running instance of a Docker image — isolated process with its own filesystem, network, and process space.
  • docker run [image]: Creates AND starts a container from an image. e.g., `docker run -d -p 80:80 nginx` runs Nginx in background on port 80.
  • docker ps: Lists running containers. docker ps -a: Lists ALL containers including stopped ones.
  • docker stop [container]: Gracefully stops a running container (sends SIGTERM). docker kill [container]: Forcefully kills it (SIGKILL).
  • docker rm [container]: Removes a stopped container. docker rm -f: Force removes even running containers.
  • docker exec -it [container] bash: Opens an interactive shell inside a running container — great for debugging.
  • -d flag: Detached mode, runs container in background. -p 8080:80: Maps host port 8080 to container port 80. --name myapp: Names the container.
📝 Dockerfile & Commands ▼
  • Dockerfile: A text file with instructions to build a custom Docker image automatically. `docker build -t myimage .` builds from Dockerfile.
  • FROM: The first instruction — specifies the base image. e.g., `FROM openjdk:17` starts with a Java 17 image.
  • RUN: Executes a command during image build. e.g., `RUN apt-get update && apt-get install -y curl` installs curl.
  • COPY/ADD: Copies files from your machine into the image. `COPY target/app.jar /app/app.jar`
  • ENV: Sets environment variables inside the container. `ENV JAVA_OPTS=-Xmx512m`
  • EXPOSE: Documents which port the container listens on. e.g., `EXPOSE 8080`. Doesn't actually open the port — just documentation.
  • CMD/ENTRYPOINT: Specifies the command to run when a container starts. `CMD ["java", "-jar", "/app/app.jar"]`
🎼 Docker Compose ▼
  • Docker Compose lets you define and run multi-container applications using a single YAML file (docker-compose.yml).
  • Instead of running 3 separate `docker run` commands for your app, database, and cache — define all 3 in one YAML and start everything with `docker-compose up`.
  • docker-compose up: Starts all services defined in docker-compose.yml. docker-compose down: Stops and removes all containers.
  • Services: Each container in Compose is a "service" — you define image, ports, environment variables, volumes, and dependencies.
  • depends_on: Ensures one service starts before another. e.g., your app service `depends_on: db` — database starts first.
  • Networks: Compose automatically creates a network so all services can communicate using service names as hostnames.
  • docker-compose logs: View logs from all services. docker-compose ps: Shows status of all services. `docker-compose up --build`: Forces rebuild of images.
🌐 Docker Hub & Docker Trusted Registry ▼
  • Docker Hub (hub.docker.com): The default public registry — like GitHub but for Docker images. Millions of public images are available here.
  • Official Images: Images maintained by Docker or the software vendor (e.g., `nginx`, `mysql`, `node`, `python`) — safe to use, regularly updated.
  • docker push [image]: Uploads your local image to Docker Hub so others can download and use it.
  • docker pull [image]: Downloads an image from Docker Hub to your local machine.
  • Image naming: `username/image-name:tag` e.g., `mycompany/webapp:v1.0`. Default tag is `latest` if not specified.
  • Docker Trusted Registry (DTR): Docker's enterprise-grade private registry — host it within your company firewall for security and control over images.
  • DTR provides role-based access control, image signing (security), vulnerability scanning, and audit logging — features Docker Hub doesn't offer for free.
🐝 Docker Swarm ▼
  • Docker Swarm is Docker's native clustering and orchestration tool — runs and manages containers across multiple machines (a "swarm" of nodes).
  • It turns a group of Docker hosts into a single virtual Docker host — you deploy once, Swarm handles the distribution.
  • Manager Node: Controls the swarm, accepts deploy commands, and distributes work to worker nodes.
  • Worker Node: Actually runs the containers assigned by the manager. Workers don't make scheduling decisions.
  • Services in Swarm: Instead of running containers, you deploy "services" — Swarm ensures the desired number of replicas are always running.
  • Self-healing: If a container crashes, Swarm automatically starts a replacement on another node — zero manual intervention.
  • Docker Swarm vs Kubernetes: Swarm is simpler and built into Docker. Kubernetes (K8s) is more powerful and the industry standard for large-scale production.
🔌 Docker Attach & CLI Deep Dive ▼
  • docker attach [container]: Connects your terminal to a running container's main process — you see its live output and can send input.
  • Difference from exec: `attach` connects to the MAIN process (PID 1). `exec` runs a NEW process inside the container (e.g., a new bash shell).
  • CTRL+P then CTRL+Q: Detach from an attached container WITHOUT stopping it — leaves it running in background.
  • docker logs [container]: View stdout/stderr output from a container without attaching. `-f` flag follows logs in real-time like `tail -f`.
  • docker inspect [container]: Returns detailed JSON with all container settings — IP address, mount points, environment variables, network config.
  • docker stats: Real-time monitoring — shows CPU, memory, network I/O, and disk I/O for all running containers.
  • docker cp [container]:[path] [local-path]: Copies files between a container and your local filesystem — useful for extracting logs or configs.
